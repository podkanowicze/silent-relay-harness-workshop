Your loop

Read → infer → prompt → inspect → pass

  1. ReadDelta, preview, code, last reply.
  2. InferWhat is this product actually for?
  3. PromptGive relevant context; request one feature.
  4. InspectWatch activity. Check the visible result.
  5. PassLeave code and a useful final reply.
2 min. Participants may prompt repeatedly. The timer does not reset and does not disable actions.

A useful turn prompt

Context is not scope.

PRODUCT

This is an internal tool for…

INVARIANT

It must continue to…

CURRENT DELTA

For this turn, add only…

STOP CONDITION

Stop once the visible behavior…

This is an internal allocation tool, not a customer checkout.
Preserve the existing approval boundary.
For this turn, add only the requested status filter.
Stop when it works in the current list.
The example is deliberately domain-neutral so it does not spoil any exercise.

Scope discipline

One turn is not permission
to finish the application.

1

One small, independently testable feature.

≠

A full specification is context, not a checklist.

Δ

Prefer the smallest viable diff.

□

Leave adjacent workflows untouched.

This is also enforced in the host policy. Mention it because zero-shotting the whole product destroys the experiment.

Legal memory

Make intent discoverable,
not hidden.

Strong carriers

  • Visible labels and behavior
  • Meaningful data structures
  • Domain-specific function names
  • Concise final reply

Shortcuts

  • New documentation files
  • Full spec in comments
  • Invisible DOM or encoded text
  • “Implement the next step”
Strong context is part of the product state. Hidden handoff payloads defeat the purpose.
